## Problem Statement

用戶在多個工作區使用 Claude Code 管理非軟體開發專案（SEO 優化、食品展、社群經營等）。每個 session 之間存在知識斷層——上一個 session 知道的工具、方法、已知的坑，下一個 session 不知道，導致重複踩坑、被用戶反覆提醒。

根因：工作模式和設定散落在各工作區，有些有完整 hook 體系（食品展），有些完全沒有（SEO 優化）。沒有統一的機制確保 session 銜接品質。

## What Makes This Cool

一個給**非工程師**用的 Claude Code plugin。現有的 plugin（gstack、sd0x-dev-flow、superpowers）都是給軟體開發者設計的——code review、TDD、precommit。但用 Claude Code 管專案的人需要的是：story 卡片管理、session handoff 品質保證、自動健康檢查、定期工作紀錄。

這在 Claude Code plugin 生態裡是空白——沒有人做「專案管理模式」的 plugin。

## Constraints

- 用戶非全職工程師，plugin 結構要能用 Claude Code 自己幫忙建
- 必須跟現有 plugin（sd0x-dev-flow、superpowers）共存，不衝突
- 食品展工作區已有可驗證的 hook 腳本（file-audit.sh、data-check.sh），可作為起點
- 先自用驗證，有效果再考慮開源

## Premises

1. 核心痛點是「session 間的知識斷層」，不是工具缺乏——工具都在（memory、handoff、story），問題是沒有機制強制使用
2. 現有 plugin 解決工程師問題，這個 plugin 解決「用 Claude Code 管專案」的問題
3. 做成 plugin 是為了一次設定、全工作區生效，且保留開源路徑

## Approaches Considered

### Approach A: Personal Plugin（選定）
做成 Claude Code plugin，有 hooks.json、scripts/、skills/。裝上就全工作區生效。
- Effort: M (human: ~2 days / CC: ~30 min)
- Risk: Med
- Pros: 一次裝、全生效、可版本控制、未來可上 marketplace
- Cons: 需要學 plugin 結構、專案特定設定還是要各工作區自己配

### Approach B: Global Hooks + Config Repo
全域 settings.json 加 hooks，腳本放 ~/.claude/hooks/。
- Effort: S (human: ~4 hours / CC: ~15 min)
- Risk: Low
- Pros: 最快能用、簡單直接
- Cons: 沒有自動安裝/更新機制、開源路徑不明確

### Approach C: Plugin + Project Templates（Phase 2 擴充）
A 的基礎上加「專案模板」機制，/init 問專案類型自動套用。
- Effort: L (human: ~1 week / CC: ~1 hour)
- Risk: Med
- Pros: 最完整、新工作區體驗最好
- Cons: 工程量最大、要維護多套模板

## Recommended Approach

**Approach A: Personal Plugin**。先驗證核心機制（hooks + session 管理），確認有效後再加 C 的模板機制。A 是 C 的子集，不走彎路。

## Plugin 架構設計

```
~/.claude/plugins/marketplaces/{plugin-name}/
├── .claude-plugin/
│   └── plugin.json          # plugin 元數據
├── hooks/
│   └── hooks.json           # SessionStart + Stop hooks
├── scripts/
│   ├── session-start.sh     # 讀 handoff + story + 開工前置 checklist
│   ├── stop-guard.sh        # 檢查 handoff 寫了沒、格式對不對
│   └── file-audit-core.sh   # 通用版：根目錄散落檔、工作紀錄連續性
├── skills/
│   ├── init/SKILL.md        # /init — 初始化工作區（產生 project config）
│   ├── handoff/SKILL.md     # /handoff — 產生標準格式 handoff
│   └── standup/SKILL.md     # /standup — 每日站會（讀 story + 進度）
└── README.md
```

### hooks.json

```json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "bash ${CLAUDE_PLUGIN_ROOT}/scripts/session-start.sh"
      }]
    }],
    "Stop": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "bash ${CLAUDE_PLUGIN_ROOT}/scripts/stop-guard.sh"
      }]
    }]
  }
}
```

### session-start.sh 核心邏輯

1. 找最新的 handoff 文件（`文檔/session-handoff-*.md` 或 `00_專案管理/工作紀錄/*.md`）
2. 找活躍的 story 卡片（狀態不是「已完成」的）
3. 輸出開工提醒：上次做到哪 + 下一步 + 需要的工具/skill
4. 如果 handoff 有「開工前置」區塊，列出 checklist

### stop-guard.sh 核心邏輯

1. 檢查今天有沒有寫 handoff
2. 如果有，檢查是否包含「開工前置」區塊
3. 檢查活躍 story 的子任務是否有更新
4. warn mode（預設）或 strict mode（opt-in）

### 專案配置機制

每個工作區放一個 `.claude/project-config.json`：

```json
{
  "project_type": "seo",
  "handoff_dir": "文檔",
  "handoff_pattern": "session-handoff-*.md",
  "story_dir": "文檔/stories",
  "work_log_dir": "文檔",
  "work_log_format": "session-handoff-{date}.md"
}
```

腳本讀這個檔案決定去哪找 handoff、story、工作紀錄。沒有這個檔案就用預設值。

## Story 卡片管理機制

Plugin 內建 story 卡片格式（參考 SEO 優化工作區的實踐）：

### Story 模板

```markdown
# {ID} {標題}

| 欄位 | 值 |
|------|------|
| 優先級 | 🔴 高 / 🟡 中 / 🟢 低 |
| 狀態 | 規劃中 / 進行中 / ✅ 已完成 / ⛔ 受阻 |
| 建立日 | {YYYY-MM-DD} |
| 完成日 | {YYYY-MM-DD} |
| 負責人 | {name} |
| 工作線 | {line} |

## 背景
{為什麼要做這件事}

## 前置確認
- [ ] {開工前需要確認的資訊}

## 子任務
- [ ] {具體步驟}

## 驗收條件
1. {可驗證的完成標準}

## 結案紀錄
{完成後補填：關鍵決策、產出物位置、技術備註}
```

### Skills 整合

- **/story-new**：建立新 story（自動編號、套模板、放到 story_dir）
- **/story-list**：列出所有活躍 story（非「已完成」），按優先級排序
- **/story-close {ID}**：結案（補填完成日 + 結案紀錄，子任務全勾）

### SessionStart hook 整合

session-start.sh 讀 story_dir，列出所有活躍 story 摘要：
```
📋 活躍 Story：
  🔴 S002 西班牙檸檬汁桶裝上架 — 進行中（Step 1）
  🟡 S011 分類頁 Meta Description — 規劃中
  🟡 S013 工作設定模組化 — 規劃中
```

### Stop hook 整合

stop-guard.sh 檢查：如果本次 session 有操作某個 story 的相關檔案，但 story 子任務沒有更新，提醒更新。

## Open Questions

1. Plugin 命名？（暫定 `pm-flow` 或 `project-flow`）
2. 跟 sd0x-dev-flow 的 Stop guard 會不會衝突？（兩個都攔 Stop event）
3. handoff「開工前置」區塊的必填項目要定到多細？
4. 是否需要 /standup skill？還是 SessionStart hook 就夠了？

## Success Criteria

1. 在 SEO 優化工作區安裝後，連續 3 個 session 不再被用戶提醒「已有的工具」
2. Handoff 品質穩定——每份 handoff 都有「開工前置」區塊
3. 食品展工作區也能用同一個 plugin（取代現有散落腳本）
4. 新工作區 10 分鐘內設定完成

## What I noticed about how you think

- 你說「我覺得這樣很亂」的時候，不是在抱怨——你在描述一個系統性問題。你的本能是把混亂收斂成結構。
- 你選「先自用，有效再開源」而不是一開始就想做大，這跟你寫文案時的風格一致——「弱者慕強，強者慕真」，先確認真的有效。
- 你提到「參考敏捷專案管理」但沒有說要照搬 Scrum 或 Kanban——你要的是敏捷的精神（sprint、standup、retrospective），不是儀式。這個判斷很重要，因為大部分「敏捷工具」都死在儀式太重。
- 你說「比較像是給非 code 專業的專案管理方式」——這是一個市場定位，不只是功能描述。Claude Code 的用戶正在從工程師擴展到各種專業人士，但工具鏈還沒跟上。
